# 岗位匹配模块 — 设计与实现

## 一、概述

### 1.1 功能说明

基于 **Milvus 向量数据库** 实现简历-岗位智能匹配：求职者上传简历后，系统根据简历内容与数据库中所有岗位进行向量相似度匹配，推荐最合适的岗位。

**核心流程**：简历 → AI 生成摘要 → DashScope Embedding 转向量 → Milvus ANN 搜索 → MySQL 补全详情 → 返回匹配结果

### 1.2 技术栈

| 层 | 技术 |
|---|---|
| 向量数据库 | Milvus 2.5.x（Standalone 模式） |
| Java SDK | milvus-sdk-java 3.0.2（MilvusClientV2） |
| Embedding | 阿里云通义 text-embedding-v3（1024 维） |
| AI 摘要 | MiMo 大模型（生成简历/岗位结构化摘要） |
| 后端 | Spring Boot 3.2.5 + MyBatis-Plus |

## 二、架构设计

## 三、Milvus 集合设计

### 3.1 集合名：`job_vectors`

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | Int64 (PK) | 对应 MySQL job.id，autoID=false |
| `vector` | FloatVector(1024) | 岗位摘要的 embedding 向量 |
| `job_id` | Int64 | 冗余，方便过滤 |
| `category_id` | Int64 | 岗位分类 ID，用于标量过滤 |
| `city` | VarChar(50) | 城市，用于标量过滤 |
| `status` | VarChar(20) | 岗位状态，只索引 active |

**索引**：IVF_FLAT，metricType=COSINE，nlist=128

**设计说明**：
- 不单独建 resume 集合——简历是实时查询的，直接调 embedding 搜 job_vectors 即可
- `job_id` 冗余字段方便过滤表达式使用
- `status` 字段确保只搜索 active 状态的岗位

## 四、后端设计

### 4.1 配置层

**application.yml** 新增：

```yaml
# Milvus 配置
milvus:
  uri: http://localhost:19530
  collection: job_vectors
  dimension: 1024

# DashScope Embedding 配置（阿里云通义）
dashscope:
  api-key: your-dashscope-api-key
  model: text-embedding-v3
  dimension: 1024
```

### 4.2 EmbeddingService — 文本转向量

```java
public interface EmbeddingService {
    /** 单条文本转向量 */
    List<Float> embed(String text);
    /** 批量文本转向量 */
    List<List<Float>> embedBatch(List<String> texts);
}
```

实现：调用 DashScope OpenAI 兼容格式 `POST /compatible-mode/v1/embeddings`
- 用 RestTemplate 直接调用，不引入额外 SDK
- 模型：`text-embedding-v3`，维度 1024
- 支持批量，每次最多 6 条

### 4.3 MilvusService — 集合管理

```java
public interface MilvusService {
    void initCollection();
    void upsertJobVectors(List<Long> jobIds, List<List<Float>> vectors,
                          List<Long> categoryIds, List<String> cities, List<String> statuses);
    void deleteJobVectors(List<Long> jobIds);
    List<Long> searchSimilarJobs(List<Float> queryVector, Map<String, String> filters, int topK);
}
```

使用 Milvus SDK 3.0.x 的 `MilvusClientV2`，API 更简洁：
- 无 `R<>` 包装器，方法直接抛异常
- `SearchResp.SearchResult` 直接返回 `entity` Map 和 `score`
- 请求/响应类都在 `io.milvus.v2.*` 包下

### 4.4 AIService — 摘要生成

```java
/** 生成简历摘要（用于 embedding） */
String generateResumeSummary(Resume resume);
/** 生成岗位摘要（用于 embedding） */
String generateJobSummary(Job job);
```

Prompt 策略：
- **简历摘要**：提取核心技能、工作经验年限、技术栈、求职意向，输出 200-300 字结构化摘要
- **岗位摘要**：提取核心技术要求、经验要求、岗位职责要点，输出 200-300 字结构化摘要
- 用 MiMo 模型，maxTokens=512

### 4.5 JobMatchService — 匹配核心

```java
public interface JobMatchService {
    List<JobMatchResult> matchJobs(Long resumeId, int topN, Map<String, String> filters);
    void syncJobToMilvus(Long jobId);
    void syncAllJobsToMilvus();
    void deleteJobFromMilvus(Long jobId);
}
```

匹配流程：
1. 从 MySQL 读取简历 → AI 生成摘要 → embedding 转向量
2. 调用 `MilvusService.searchSimilarJobs()` 做 ANN 搜索
3. 用 MySQL 补全岗位详情（公司名、分类名等）
4. 返回 `List<JobMatchResult>`

### 4.6 JobMatchController — API 接口

```
POST /api/job-match/resume/{resumeId}
  参数: topN (默认10), city, categoryId
  返回: { matches: [...], total: N }

POST /api/job-match/sync/{jobId}       // 岗位变更时手动同步
POST /api/job-match/sync-all            // 全量同步（管理员）
POST /api/job-match/delete/{jobId}      // 从 Milvus 删除
```

## 六、容错设计

### 6.1 Milvus 不可用时的降级

`MilvusServiceImpl.init()` 中 `@PostConstruct` 捕获异常并 warn 日志，不影响应用启动。匹配接口在 Milvus 不可用时抛出运行时异常，由全局异常处理器返回错误信息。

### 6.2 AI 摘要生成失败

`AIServiceImpl` 中摘要生成失败时返回空字符串，embedding 服务会对空文本生成默认向量，搜索结果可能不准确但不会报错。

### 6.3 Embedding 维度不匹配

确保 DashScope `text-embedding-v3` 配置维度（1024）与 Milvus 集合维度一致。

## 八、部署

### 8.1 Docker Compose 部署（推荐）

项目根目录 `docker-compose.yml` 已包含 Milvus 全套服务：

```bash
# 启动 Milvus 及其依赖（etcd + minio）
docker compose up -d etcd milvus-minio milvus

# 查看状态
docker compose ps

# 查看日志
docker compose logs -f milvus
```

服务说明：

| 服务 | 端口 | 说明 |
|------|------|------|
| etcd | — | Milvus 元数据存储 |
| milvus-minio | 9100/9101 | Milvus 对象存储（避免与主 MinIO 冲突） |
| milvus | 19530 (gRPC), 9091 (健康检查) | Milvus 向量数据库 |

### 8.2 单独部署 Milvus（不依赖 docker-compose）

如果只需要 Milvus 而不想启动整个 docker-compose 栈，可以单独启动。

三个容器通过容器名互相通信（`milvus-etcd:2379`、`milvus-minio:9000`），必须加入同一网络。使用项目已有的 `junle` 网络：

```bash
# 1. 启动 etcd
docker run -d --name milvus-etcd \
  --network junle \
  -e ETCD_AUTO_COMPACTION_MODE=revision \
  -e ETCD_AUTO_COMPACTION_RETENTION="1000" \
  -e ETCD_QUOTA_BACKEND_BYTES="4294967296" \
  -e ETCD_SNAPSHOT_COUNT="50000" \
  -v etcd_data:/etcd \
  quay.io/coreos/etcd:v3.5.25 \
  etcd -advertise-client-urls=http://127.0.0.1:2379 -listen-client-urls http://0.0.0.0:2379 --data-dir /etcd

# 2. 启动 MinIO（Milvus 专用）
docker run -d --name milvus-minio \
  --network junle \
  -e MINIO_ACCESS_KEY=minioadmin \
  -e MINIO_SECRET_KEY=minioadmin123 \
  -p 9100:9000 -p 9101:9001 \
  -v milvus_minio_data:/minio_data \
  minio/minio:RELEASE.2024-05-28T17-19-04Z \
  minio server /minio_data --console-address ":9001"

# 3. 启动 Milvus（依赖 etcd 和 minio，通过容器名访问）
docker run -d --name milvus \
  --network junle \
  -p 19530:19530 -p 9091:9091 \
  -e ETCD_ENDPOINTS=milvus-etcd:2379 \
  -e MINIO_ADDRESS=milvus-minio:9000 \
  -v milvus_data:/var/lib/milvus \
  milvusdb/milvus:v2.4.5 \
  milvus run standalone

# 4. 验证服务健康
curl http://localhost:9091/healthz
```

> **清理命令**（不再需要时）：
> ```bash
> docker stop milvus milvus-minio milvus-etcd
> docker rm milvus milvus-minio milvus-etcd
> docker volume rm etcd_data milvus_minio_data milvus_data
> ```

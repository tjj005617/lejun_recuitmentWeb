# 知识图谱模块 — 设计与实现

## 一、概述

### 1.1 这个模块用来干什么

知识图谱模块是乐俊招聘平台**八股面试专栏**的核心功能，解决的问题是：

> 面试准备时，知识点散落在各种文档中，缺乏关联。求职者不知道"学 JVM 之前要先学什么"、"Spring 和 Spring Boot 是什么关系"、"GC 和内存模型怎么联系"。本模块通过 AI 自动从技术文档中提取知识点及其关联关系，构建可视化的知识图谱，让求职者能够**系统化、结构化地学习面试知识**。

### 1.2 做什么功能

#### 管理员侧（后台）

| 功能 | 说明 |
|---|---|
| 上传文档 | 上传八股文 Markdown/TXT 文档，AI 自动分类（或手动指定分类） |
| AI 自动处理 | 无需人工干预：AI 自动分类 → 提取知识点 → 分析关系 → 构建缓存 |
| 实时进度 | SSE 实时推送处理进度，进度条平滑推进，失败可重试 |
| 文档管理 | 查看文档列表、处理状态、错误信息；支持删除（级联清除图谱数据） |

#### 用户侧（前台）

| 功能 | 说明 |
|---|---|
| 分类浏览 | 按技术领域（Java/前端/数据库等）切换知识图谱 |
| 图谱可视化 | ECharts 力导向图展示知识点网络，节点大小反映关联密度 |
| 搜索知识点 | 关键词搜索，快速定位目标知识点 |
| 查看详情 | 点击节点查看：类型、难度、重要性、描述、标签、来源文档 |
| 关联导航 | 查看一跳邻居，点击邻居跳转详情，连续浏览知识网络 |
| 列表视图 | 卡片网格展示所有知识点，适合概览和快速查找 |

### 1.3 怎么使用

#### 管理员操作流程

```
1. 登录管理后台 → 进入"知识图谱管理"页面
2. 点击"上传文档" → 填写标题（可选）→ 选择本地 MD/TXT 文件
3. 点击"上传并处理" → 弹窗切换到进度视图
4. 等待 AI 处理（约 60-90 秒），观察进度条：
   - 解析文档 → AI 提取知识点 → AI 分析关系 → 构建图谱
5. 处理完成 → 点击"完成"关闭弹窗 → 表格中查看结果
6. 如处理失败 → 查看错误信息 → 点击"重试"重新处理
7. 点击"查看图谱" → 跳转到前台图谱页面预览效果
```

#### 用户使用流程

```
1. 访问"八股面试专栏"页面（/knowledge）
2. 页面默认加载第一个分类的图谱（如 Java）
3. 点击顶部分类标签切换不同技术领域
4. 在图谱视图中：
   - 拖拽画布平移（手型模式）
   - 滚轮缩放图谱
   - 悬停节点高亮关联关系
   - 点击节点打开右侧详情抽屉
5. 在详情抽屉中：
   - 查看知识点类型、难度、重要性、描述、标签
   - 浏览关联知识点列表
   - 点击关联知识点跳转查看详情
6. 切换到列表视图，以卡片形式浏览所有知识点
7. 在搜索框输入关键词，快速搜索知识点
```

### 1.4 核心能力

- **AI 自动处理**：上传 MD/TXT 文档 → 自动分类 → 提取知识点 → 分析关系 → 构建邻接缓存
- **双模型协作**：MiMo 负责分类和知识点提取，DeepSeek 负责关系分析
- **实时进度推送**：SSE（Server-Sent Events）实时推送处理进度到管理后台
- **可视化交互**：ECharts 力导向图渲染，支持拖拽、缩放、点击查看详情
- **知识导航**：通过关联关系连续浏览，构建知识网络的心智模型

### 1.5 技术栈

| 层 | 技术 |
|---|---|
| 前端 | Vue 3 + Element Plus + ECharts（force-directed graph） |
| 后端 | Spring Boot + MyBatis-Plus + @Async 异步处理 |
| AI | MiMo（分类+提取）、DeepSeek（关系分析） |
| 存储 | MySQL（图谱数据）、MinIO（文档文件）、SSE（进度推送） |

**edge_label 枚举：**

| 值 | 含义 | 典型场景 |
|---|---|---|
| prerequisite | 前置知识 | 学习 GC 前需要先了解 JVM |

| 范围 | 含义 |
|---|---|
| 1.00 | 文档中明确陈述的关系 |
| 0.70 - 0.99 | 强烈暗示的关系 |
| 0.40 - 0.69 | 基于通用知识的推断 |

### 2.5 kg_adjacency（邻接缓存）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | INTEGER | PK AUTO_INCREMENT | 主键 |
| vertex_id | INTEGER | NOT NULL, INDEX | 知识点ID |
| neighbor_id | INTEGER | NOT NULL, INDEX | 邻居知识点ID |
| direct_label | VARCHAR(50) | NOT NULL | 关系标签 |
| direction | VARCHAR(10) | NOT NULL | outgoing / incoming |
| weight | DECIMAL(3,2) | DEFAULT 0.50 | 关系权重 |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | |
| | | UNIQUE(vertex_id, neighbor_id, direct_label) | |

**设计说明：** 邻接缓存是图谱查询性能的关键。查询一跳邻居时直接走 `kg_adjacency` 表的索引，避免了 JOIN `kg_edge` 的两次索引查找。文档处理流水线的最后一步自动构建此缓存。

### 4.2 核心接口详细设计

#### 上传文档

```
POST /api/admin/kg/upload
Content-Type: multipart/form-data
```

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| file | File | 是 | .md / .txt / .markdown |
| categoryName | String | 否 | 分类名称（空则 AI 自动分类） |
| title | String | 否 | 文档标题（空则用文件名） |

响应返回文档对象（含 id），前端据此建立 SSE 连接。

## 六、后端设计

### 6.1 处理流水线（KgProcessingServiceImpl）

核心方法 `processDocument(Long documentId)` 通过 `@Async("kgExecutor")` 在独立线程池中执行。

```
processDocument(documentId)
  │
  ├── Step 1: 解析 + 自动分类 (10% → 20%)
  │   ├── 读取文档 raw_content
  │   ├── 如果 category_id 为空：
  │   │   └── AI.classifyCategory(content, categories)  ← MiMo, maxTokens=64
  │   └── 更新 category_id
  │
  ├── Step 2: 提取知识点 (30% → 60%)
  │   ├── AI.extractVertices(content, categoryName)  ← MiMo, maxTokens=4096
  │   │   └── 返回 15-40 个 KgVertex
  │   ├── 逐个持久化顶点（设 documentId, categoryId, UUID）
  │   └── 更新 document.vertex_count
  │
  ├── Step 3: 分析关系 (65% → 85%)
  │   ├── AI.analyzeEdges(vertices, content, categoryName)  ← DeepSeek, maxTokens=4096
  │   │   └── AI 返回 from_name/to_name → 解析为数据库 ID
  │   ├── 过滤无效边（null ID、自环）
  │   ├── 逐个持久化边（唯一键冲突静默跳过）
  │   └── 更新 document.edge_count
  │
  ├── Step 4: 构建邻接缓存 (90% → 100%)
  │   └── 对每个顶点：
  │       ├── 查询出边 → upsert adjacency(outgoing)
  │       └── 查询入边 → upsert adjacency(incoming)
  │
  └── 完成: status → completed, current_step → null
```

### 6.3 SSE 进度推送机制

```
KgSseManager
  │
  ├── ConcurrentHashMap<Long, SseEmitter>
  │   key = documentId
  │   value = SseEmitter (无超时)
  │
  ├── addEmitter(docId, emitter)
  │   └── 如果已有旧 emitter → 先 complete 关闭
  │
  ├── sendProgress(docId, data)
  │   └── emitter.send(SseEmitter.event().name("kg_progress").data(json))
  │       └── IOException → removeEmitterIfCurrent()
  │
  └── removeEmitterIfCurrent(docId, emitter)
      └── 只有 emitter 实例匹配时才移除（防止旧回调误删新连接）
```

**为什么独立于消息 SSE：**
- 管理员使用 `admin_token`（localStorage），普通用户使用 `token`
- KG 进度按 `documentId` 隔离，消息通知按 `userId` 隔离
- 两者关注点不同，不应复用同一连接

### 6.4 线程池配置

```java
@Bean("kgExecutor")
public Executor kgExecutor() {
    return Executors.newFixedThreadPool(3);  // 3 线程处理 KG 文档
}
```

选择 3 线程的原因：
- AI 调用是 I/O 密集型，3 线程可并发处理 3 个文档
- 每个文档处理约 60-90 秒（3 次 AI 调用 + 数据库写入）
- 避免过多并发导致 AI API 限流

#### 7.2.2 核心功能模块

**① 分类导航**

- 页面挂载时调用 `getKgCategories()` 加载启用分类列表
- 分类栏使用 `position: sticky; top: 64px` 吸顶，滚动时不丢失
- 点击分类 → `selectCategory(id)` → 更新路由 `/knowledge/{id}` → 加载图谱数据
- 路由参数变化通过 `watch(() => route.params.categoryId)` 监听，支持浏览器前进/后退
- 默认加载第一个分类

**② 搜索功能**

- Banner 内嵌搜索框，回车触发 `handleSearch()`
- 调用 `searchKgVertices(keyword, categoryId)`，支持按当前分类过滤
- 搜索结果以卡片列表展示，显示"搜索「xxx」找到 N 个知识点"
- 提供"清除搜索"按钮回到分类图谱视图
- 搜索状态下隐藏图谱/列表切换

**③ 视图切换**

| 模式 | 说明 |
|---|---|
| 图谱视图 | ECharts 力导向图，roam=true 支持拖拽+缩放 |
| 列表视图 | 卡片网格，展示所有知识点的类型、名称、描述、关联数 |

切换时使用 `v-show` 保留 DOM 状态，避免重新渲染。

**④ 交互模式（仅图谱视图）**

| 模式 | 图标 | 行为 |
|---|---|---|
| 指针模式 | Pointer | 点击节点打开详情，画布不可拖拽 (`roam: 'move'`) |
| 手型模式 | Position | 拖拽画布平移，光标变为 grab/grabbing (`roam: true`) |

通过 `interactionMode` ref 切换，动态调用 `chartInstance.setOption()` 修改 roam 属性。

**⑤ Ctrl+滚轮缩放**

在 `.kg-graph-container` 上监听 `wheel` 事件（`passive: false`），当检测到 `e.ctrlKey` 时调用 `preventDefault()` + `stopPropagation()` 阻止浏览器页面缩放，仅保留 ECharts 图谱缩放。

绑定范围：外层 `.kg-graph-container`（非内层 chart div），避免判定范围过小导致误操作。

#### 7.2.3 ECharts 图谱配置

```javascript
{
  type: 'graph',
  layout: 'force',
  roam: true,              // 拖拽平移 + 滚轮缩放
  force: {
    repulsion: Math.max(200, nodes.length * 15),  // 动态斥力
    edgeLength: [80, 200],                         // 边长范围
    gravity: 0.1                                   // 引力
  },
  emphasis: {
    focus: 'adjacency',    // 悬停高亮关联节点
    blurScope: 'global'    // 其余节点模糊
  },
  edgeSymbol: ['none', 'arrow'],   // 有向边箭头
  edgeSymbolSize: [0, 8]
}
```

**节点渲染规则：**

| 属性 | 计算方式 |
|---|---|
| `symbolSize` | `20 + (edgeCount / maxEdges) * 40`，连接越多越大 |
| `itemStyle.color` | 按 `vertexType` 映射颜色（见 2.3 节枚举） |
| `label.show` | `edgeCount >= 2` 或 `nodes.length <= 30` 时显示 |

**边渲染规则：**

| 属性 | 计算方式 |
|---|---|
| `lineStyle.width` | `max(1, weight * 3)`，权重越大线越粗 |
| `lineStyle.curveness` | `0.2`，微弯曲避免重叠 |
| `emphasis.lineStyle` | 悬停时变蓝 + 加粗至 3px |
| `emphasis.label.show` | 悬停时显示关系标签 |

**Tooltip（悬停提示）：**

悬停节点显示：名称、分类（绿色）、类型（彩色）、描述、关联数。悬停边显示关系标签。

```javascript
const connectKgSSE = (documentId) => {
  const token = localStorage.getItem('admin_token')
  const es = new EventSource(
    `${apiBase}/admin/kg/subscribe?token=${token}&documentId=${documentId}`
  )

  es.addEventListener('kg_progress', (e) => {
    const data = JSON.parse(e.data)
    processingStep.value = data.step
    processingPercent.value = data.progress

    if (data.status === 'completed') {
      processingResult.value = 'completed'
      processingPercent.value = 100
      disconnectKgSSE()
    } else if (data.status === 'failed') {
      processingResult.value = 'failed'
      processingErrorMsg.value = data.errorMessage
      disconnectKgSSE()
    }
  })

  es.onerror = () => disconnectKgSSE()
  kgEventSource = es
}
```

## 八、关键设计决策

### 8.1 为什么用双模型？

| 任务 | 模型 | 原因 |
|---|---|---|
| 分类 | MiMo | 轻量快速，64 tokens 即可 |
| 知识点提取 | MiMo | 中文理解能力强，提取质量高 |
| 关系分析 | DeepSeek | 逻辑推理能力强，关系判断更准确 |

### 8.2 为什么需要邻接缓存？

查询一跳邻居的两种方式对比：

| 方式 | SQL | 耗时 |
|---|---|---|
| 直接查 kg_edge | `SELECT * FROM kg_edge WHERE from_id=? OR to_id=?` | ~50ms（两次索引扫描） |
| 查 kg_adjacency | `SELECT * FROM kg_adjacency WHERE vertex_id=?` | ~5ms（单次索引命中） |

邻接缓存用空间换时间，文档处理时一次性构建，查询时 O(1) 索引命中。

### 8.3 为什么 SSE 而不是轮询？

| 方式 | 延迟 | 服务器压力 | 用户体验 |
|---|---|---|---|
| 3秒轮询 | 0-3秒 | 每3秒1次请求 | 进度跳跃 |
| SSE | <100ms | 1次长连接 | 进度平滑 |

SSE 在文档处理场景下优势明显：处理耗时 60-90 秒，轮询约产生 20-30 次无效请求，SSE 只有 1 次连接。

### 8.4 为什么移除 draggable？

ECharts graph 的 `draggable: true`（节点拖拽）和 `roam: true`（画布拖拽）会冲突：
- `draggable: true` 拦截鼠标事件用于拖动单个节点
- 导致画布级别的平移（pan）无法响应

解决方案：移除 `draggable`，仅保留 `roam: true`，用户通过拖拽画布来平移图谱。
